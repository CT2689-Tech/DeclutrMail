## DQ5 — the one real product challenge

### D-CHALLENGE: queued-draft mailto unsubscribe [blocks: newsletter-cleanup magic moment | reversible: yes]

**Current decision:** D230 — mailto unsubscribe is manual at launch: per-sender "opens in Gmail" compose deep-link, user hits Send. Only RFC 8058 one-click executes automatically.
**Why suboptimal for $10K MRR:** Promotions cleanup is ~93% one-click (unaffected), but classic newsletter cleanup (updates category) routes **>half of actionable senders** through open-compose→switch-tab→Send, per sender. That's the aha-moment surface for the exact user we monetize.
**Proposed change:** Queued-draft + bulk-approve: create Gmail drafts (from the user's own address, `gmail.modify` already granted), show a batch approval screen, send only on explicit user approval. No auto-send from no-reply; D226 preview semantics preserved.
**Tradeoff / risk:** (1) Retires the trust copy "DeclutrMail never sends it for you" (unsub-mailto-callout.tsx:59) — touches the privacy wedge's _adjacent_ messaging; (2) a sent email is un-undoable — new UX class vs the undo-everywhere posture; (3) bulk-send of N unsub mails is a new abuse/deliverability surface.
**Guardrail check:** Does not weaken §2 privacy rules (no body storage, no new scopes). It DOES reverse D230's launch posture, so it needs an explicit D-patch if approved.
**Recommendation:** Keep D230 manual at launch; build queued-draft as the first post-beta experiment gated on funnel data (unsub-flow drop-off in PostHog). Confidence 65 — the hit is real but unquantified; the trust copy is worth more than a data-free bet pre-launch.
